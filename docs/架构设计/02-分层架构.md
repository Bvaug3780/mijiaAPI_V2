# 分层架构详解

## 架构概述

本项目采用经典的分层架构模式，从上到下分为五层：

```
API客户端层 → 服务层 → 仓储层 → 基础设施层 → 领域层
```

每一层都有明确的职责，层与层之间通过接口进行通信。

## 第一层：领域层 (Domain Layer)

### 位置
`mijiaAPI_V2/domain/`

### 职责
定义核心业务实体和业务规则，是整个系统的核心。

### 核心文件

#### 2. exceptions.py - 异常定义

**异常层次结构**：
```
MijiaAPIException (基类)
├── AuthenticationError (认证相关)
│   ├── LoginFailedError
│   ├── TokenExpiredError
│   └── InvalidCredentialError
├── DeviceError (设备相关)
│   ├── DeviceNotFoundError
│   ├── DeviceOfflineError
│   └── PropertyReadOnlyError
├── NetworkError (网络相关)
│   ├── TimeoutError
│   └── ConnectionError

```

**设计原因**：
- 清晰的异常层次便于错误处理
- 每个异常携带详细的上下文信息
- 使用中文错误消息，用户友好

### 设计原则

1. **纯粹性**：领域层不依赖任何外部框架或库（除了Pydantic）
2. **业务规则**：所有业务验证逻辑都在领域模型中
3. **不可变性**：领域对象尽可能不可变

#### 2. cache_manager.py - 缓存管理器

**三层缓存架构**：
```python
class CacheManager:
    def __init__(self, cache_dir: Path, redis_client: Optional[Redis]):
        self._memory_cache = {}      # 内存缓存（最快）
        self._redis = redis_client   # Redis缓存（分布式）
        self._file_cache_dir = cache_dir  # 文件缓存（持久化）
```

**缓存查找顺序**：
1. 内存缓存 → 命中则返回
2. Redis缓存 → 命中则更新内存缓存并返回
3. 文件缓存 → 命中则更新内存和Redis并返回
4. 未命中 → 返回None

**设计原因**：
- **性能分层**：内存最快，Redis次之，文件最慢
- **分布式支持**：Redis支持多进程/多服务器共享
- **持久化**：文件缓存在重启后仍然有效
- **命名空间隔离**：使用user_id作为命名空间，多用户互不干扰

**缓存策略**：
```python
# 设备列表：5分钟TTL（经常变化）
cache.set("devices:home_123", devices, ttl=300)

# 设备规格：1年TTL（很少变化）
cache.set("spec:model_abc", spec, ttl=365*24*3600)

# 设备状态：30秒TTL（频繁变化）
cache.set("state:device_456", state, ttl=30)
```

## 第三层：仓储层 (Repository Layer)

### 位置
`mijiaAPI_V2/repositories/`

### 职责
抽象数据访问逻辑，提供领域对象的持久化。

### 核心文件

#### 1. interfaces.py - 仓储接口

**为什么需要接口？**
- 依赖倒置：服务层依赖接口，而非具体实现
- 易于测试：可以使用Mock实现
- 易于替换：可以切换不同的数据源

**IHomeRepository**：
```python
class IHomeRepository(ABC):
    @abstractmethod
    def get_all(self, credential: Credential) -> List[Home]:
        """获取所有家庭"""
    
    @abstractmethod
    def get_by_id(self, home_id: str, credential: Credential) -> Optional[Home]:
        """根据ID获取家庭"""
```

**IDeviceRepository**：
```python
class IDeviceRepository(ABC):
    @abstractmethod
    def get_by_home(self, home_id: str, credential: Credential) -> List[Device]:
        """获取家庭下的所有设备"""
    
    @abstractmethod
    def get_by_id(self, device_id: str, credential: Credential) -> Optional[Device]:
        """根据ID获取设备"""
    
    @abstractmethod
    def set_property(self, device_id: str, siid: int, piid: int, 
                     value: Any, credential: Credential) -> bool:
        """设置设备属性"""
```

#### 2. home_repository.py - 家庭仓储实现

```python
class HomeRepositoryImpl(IHomeRepository):
    def __init__(self, http_client: HttpClient, cache_manager: CacheManager):
        self._http = http_client
        self._cache = cache_manager
    
    def get_all(self, credential: Credential) -> List[Home]:
        # 1. 检查缓存
        cached = self._cache.get("homes", namespace=credential.user_id)
        if cached:
            return [Home.model_validate(h) for h in cached]
        
        # 2. 从API获取
        response = self._http.post("/homeroom/gethome", {}, credential)
        
        # 3. 解析数据
        homes = self._parse_homes(response)
        
        # 4. 缓存结果
        self._cache.set("homes", [h.model_dump() for h in homes], 
                       ttl=300, namespace=credential.user_id)
        
        return homes
```

**设计原因**：
- 缓存优先：减少API调用
- 命名空间隔离：多用户缓存互不干扰
- 数据转换：API响应 → 领域模型

#### 4. device_spec_repository.py - 设备规格仓储

**特殊性**：
- 设备规格很少变化，使用很长的 TTL（1年）实现长期缓存
- 使用设备型号作为缓存键

```python
def get_spec(self, model: str) -> Optional[DeviceSpec]:
    # 长期缓存设备规格（1年）
    cache_key = f"spec:{model}"
    cached = self._cache.get(cache_key)
    if cached:
        return DeviceSpec.model_validate(cached)
    
    # 从API获取
    spec = self._fetch_spec(model)
    
    # 缓存1年（365天）
    self._cache.set(cache_key, spec.model_dump(), ttl=365*24*3600)
    
    return spec
```

#### 2. device_service.py - 设备服务

**核心业务逻辑**：
```python
class DeviceService:
    def get_devices(self, home_id: str, credential: Credential) -> List[Device]:
        """获取设备列表"""
        return self._device_repo.get_by_home(home_id, credential)
    
    def set_device_property(self, device_id: str, siid: int, piid: int,
                           value: Any, credential: Credential) -> bool:
        """设置设备属性（带验证）"""
        # 1. 获取设备信息
        device = self._device_repo.get_by_id(device_id, credential)
        if not device:
            raise DeviceNotFoundError(f"设备不存在: {device_id}")
        
        # 2. 获取设备规格
        spec = self._spec_repo.get_spec(device.model)
        
        # 3. 验证属性是否可写
        if not self._is_writable(spec, siid, piid):
            raise PropertyReadOnlyError(f"属性只读: siid={siid}, piid={piid}")
        
        # 4. 验证属性值范围
        if not self._validate_value(spec, siid, piid, value):
            raise ValidationError(f"属性值无效: {value}")
        
        # 5. 设置属性
        return self._device_repo.set_property(device_id, siid, piid, value, credential)
```

**设计原因**：
- 业务验证：确保操作合法
- 错误处理：提供清晰的错误信息
- 协调仓储：组合多个仓储完成复杂操作

## 第五层：API客户端层 (API Client Layer)

### 位置
`mijiaAPI_V2/api_client.py`

### 职责
对外提供统一的API接口，是用户直接使用的入口。

### 核心类

#### 1. MijiaAPI（同步）

**设计**：
```python
class MijiaAPI:
    def __init__(self, credential: Credential, 
                 device_service: DeviceService,
                 scene_service: SceneService,
                 statistics_service: StatisticsService,
                 home_repository: IHomeRepository):
        self._credential = credential
        self._device_service = device_service
        self._scene_service = scene_service
        self._statistics_service = statistics_service
        self._home_repository = home_repository
```

**为什么注入服务而非仓储？**
- 服务层已经封装了业务逻辑
- API客户端只需要调用服务方法
- 保持API简洁

**核心方法**：
```python
def get_homes(self) -> List[Home]:
    """获取家庭列表"""
    return self._home_repository.get_all(self._credential)

def get_devices(self, home_id: str) -> List[Device]:
    """获取设备列表"""
    return self._device_service.get_devices(home_id, self._credential)

def control_device(self, device_id: str, siid: int, piid: int, value: Any) -> bool:
    """控制设备"""
    return self._device_service.set_device_property(
        device_id, siid, piid, value, self._credential
    )
```

#### 2. AsyncMijiaAPI（异步）

**实现方式**：
```python
async def get_devices(self, home_id: str) -> List[Device]:
    """异步获取设备列表"""
    import asyncio
    return await asyncio.to_thread(
        self._device_service.get_devices, home_id, self._credential
    )
```

**为什么使用 asyncio.to_thread？**
- 简单实用：复用同步服务层代码
- 不阻塞事件循环：在后台线程执行
- 性能足够：对于I/O密集型操作，性能差异不大

**未来扩展**：
- 可以实现完全异步的仓储和服务层
- 使用AsyncHttpClient替代HttpClient
- 获得更细粒度的异步控制

**为什么需要工厂函数？**
- 简化对象创建
- 自动组装依赖
- 隐藏复杂的初始化逻辑

**create_api_client 实现**：
```python
def create_api_client(credential: Credential) -> MijiaAPI:
    # 1. 创建配置管理器
    config = ConfigManager()
    
    # 2. 创建基础设施组件
    crypto = CryptoService()
    http_client = HttpClient(config, crypto)

    # 3. 创建仓储层
    home_repo = HomeRepositoryImpl(http_client, cache_manager)

    scene_service = SceneService(scene_repo)
    statistics_service = StatisticsService(device_repo)
    
    # 5. 创建API客户端
    return MijiaAPI(
        credential=credential,
        device_service=device_service,
        scene_service=scene_service,
        statistics_service=statistics_service,
        home_repository=home_repo,
```

## 层间通信规则

### 1. 依赖方向
```
API客户端层 → 服务层 → 仓储层 → 基础设施层
                ↓
            领域层 ← ← ← ← ← ← ← ← ← ← ←
```

**规则**：
- 高层可以依赖低层
- 所有层都可以依赖领域层
- 低层不能依赖高层

### 2. 数据流动
```
用户请求 → API客户端 → 服务层 → 仓储层 → HTTP客户端 → 米家API
                                                    ↓
用户响应 ← API客户端 ← 服务层 ← 仓储层 ← HTTP客户端 ← 米家API
```

### 3. 异常传播
```
米家API错误 → HTTP客户端（转换为领域异常）→ 仓储层（传播）→ 服务层（处理/传播）→ API客户端（传播）→ 用户
```

## 总结

### 分层架构的优势

1. **清晰的职责划分**：每一层只关注自己的事情
2. **易于测试**：每一层可以独立测试
3. **易于维护**：修改某一层不影响其他层
4. **易于扩展**：可以轻松添加新功能
5. **易于替换**：可以替换某一层的实现

### 设计权衡

1. **复杂性 vs 灵活性**：分层增加了代码量，但提高了灵活性
2. **性能 vs 可维护性**：多层调用有性能开销，但代码更清晰
3. **抽象 vs 具体**：接口增加了抽象，但便于替换实现

### 最佳实践

1. **保持层的纯粹性**：不要跨层调用
2. **使用接口定义契约**：便于测试和替换
3. **依赖注入**：通过构造函数注入依赖
4. **单一职责**：每个类只做一件事
5. **开闭原则**：对扩展开放，对修改关闭
